# Code Graph Artifact Ingestion Spec

## Purpose

Design how a locally generated Code Graph artifact can be converted into retrievable context for the Context Engine/RAG in a future milestone.

The Code Graph artifact is derived data. It is not the primary source of truth and must not replace repository code, specs, ADRs or other source documents. Its purpose is to complement lexical, vector and hybrid retrieval with conservative structural context while preserving traceability back to repository-relative paths, symbols, edges and scanner decisions.

Future ingestion should let repo-structure questions retrieve useful context such as route-to-service relationships, test-to-source relationships, migrations, exported symbols and docs/spec/ADR links. It must keep existing data policy, retention, deletion, answerability and lexical fallback behavior intact.

## Non-Goals

Milestone 1.6E does not include:

- artifact ingestion implementation,
- database persistence,
- migrations,
- API endpoints,
- runtime integration,
- Context Engine implementation changes,
- dashboard views,
- GraphRAG,
- LLM-as-judge,
- external providers,
- real models,
- required pgvector,
- code execution,
- reading outside the repository,
- new source adapters,
- new persisted source types,
- scanner changes,
- workflow changes,
- package or dependency changes.

## Input Artifact Contract

Future ingestion must consume the JSON artifact generated by:

```bash
pnpm code-graph:scan
```

and validated by:

```bash
pnpm code-graph:check
```

The artifact is expected to contain:

- `scanRun`,
- `files`,
- `symbols`,
- `edges`,
- `warnings`.

### `scanRun`

`scanRun` identifies the scanner version, repository root, commit SHA when available, status, timestamps and scan counts. Ingestion must treat `scannerVersion`, `repoRoot`, `commitSha`, `status`, `filesScanned` and `filesSkipped` as audit metadata. Timestamps are useful for traceability but must not be used as retrieval content.

Only completed, locally generated, validated scan runs should be eligible for ingestion. A future command or service must reject artifacts whose `repoRoot` does not resolve to the active repository root or whose paths are not repository-relative.

### `files`

`files` describe repository-relative files detected by the scanner. Important fields include:

- `id`,
- `path`,
- `packageName`,
- `language`,
- `fileKind`,
- `hash`,
- `sizeBytes`,
- `isTest`,
- `isMigration`,
- `isSpec`,
- `isAdr`,
- `metadata`.

Future ingestion should use these records to create file-oriented summaries, not full-source chunks.

### `symbols`

`symbols` describe declarations and structural entities detected in files. Important fields include:

- `id`,
- `fileId`,
- `name`,
- `symbolKind`,
- `exportKind`,
- `startLine`,
- `endLine`,
- `visibility`,
- `confidence`,
- `metadata`.

Future ingestion should preserve symbol identity, kind, source file and confidence. Line numbers are navigation metadata only and should not be treated as evidence of semantic correctness.

### `edges`

`edges` describe relationships between files, symbols and documents. Important fields include:

- `id`,
- `sourceType`,
- `sourceId`,
- `targetType`,
- `targetId`,
- `edgeType`,
- `confidence`,
- `metadata`.

Current scanner edge types include:

- `imports`,
- `exports`,
- `tests`,
- `migrates`,
- `documents`.

Future ingestion should represent one clear relationship per chunk when possible, with confidence and evidence metadata available for answerability and auditing.

### `warnings`

`warnings` describe scanner omissions or unsupported patterns. Important fields include:

- `code`,
- `path`,
- `message`,
- `severity`.

Warnings are not positive evidence. Future ingestion may expose warnings as metadata or low-priority warning chunks, but warnings must not be ranked or interpreted as facts about repository behavior.

### Trust Boundary

The artifact must be local, generated inside the repository and validated before ingestion. It must not be accepted as an arbitrary external or untrusted upload without validation. Future ingestion must reject:

- paths outside the repository,
- absolute paths where repository-relative paths are required,
- symlink-derived records that point outside the repository,
- artifacts with unknown required top-level shape,
- artifacts whose scanner version is unsupported,
- artifacts that fail fixture or schema validation.

## Proposed Ingestion Model

Future implementation should treat Code Graph context as a derived Context Engine source. The conceptual source type is:

```text
code_graph
```

This spec does not add that source type to contracts or database schema. A future implementation milestone must explicitly update contracts, migrations, services, tests and retention behavior before the source type exists at runtime.

### Source

A future `code_graph` context source should represent one validated artifact for one repository state. Suggested source metadata:

```text
codeGraphScanRunId
artifactPath
commitSha
scannerVersion
filesScanned
filesSkipped
generatedFrom
```

`generatedFrom` should identify the artifact and scanner command, not a remote provider or model.

### Documents

Future ingestion should create derived `context_documents` grouped around structural concepts rather than storing the raw artifact as one large document.

Recommended document groups:

- file summaries,
- symbol summaries,
- edge summaries,
- route summaries,
- migration summaries,
- test coverage summaries,
- doc/spec/ADR relationship summaries.

Examples:

- one document for all route summaries in `apps/api`,
- one document for test coverage relationships,
- one document per file for file-level structural summaries,
- one document for docs/spec/ADR relationships.

The final grouping should be selected by evaluation quality and retention constraints, not by database convenience alone.

### Chunks

Future ingestion should create small `context_chunks` with traceable text and metadata. Chunks should contain structural summaries that are easy to retrieve and audit.

Recommended chunk metadata:

```text
codeGraphScanRunId
artifactPath
sourcePath
symbolName
symbolKind
edgeType
targetPath
confidence
generatedFrom
scannerVersion
commitSha
warningCode
```

These fields are conceptual. This milestone does not define a database schema, JSON contract or API response shape.

## Chunking Strategy

Future Code Graph chunks should be conservative:

- keep chunks small,
- prefer one relationship or one structural summary per chunk,
- avoid full source code,
- avoid duplicating whole files,
- avoid embedding large JSON arrays,
- prefer structured, retrievable sentences,
- preserve repository-relative paths and symbol names,
- preserve confidence and scanner evidence,
- do not present warnings as facts,
- do not include raw secret-like values,
- keep generated text deterministic for the same artifact and ingestion version.

Permitted example:

```text
File apps/api/src/routes/goals.ts defines Fastify route GET /api/goals and imports GoalService from apps/api/src/services/goalService.ts.
```

Not permitted:

```text
Ingest full source code from apps/api/src/routes/goals.ts as one chunk.
```

### File Summary Chunks

File summaries should describe the file role and major structural facts:

```text
File apps/api/src/services/goalService.ts is a TypeScript source file in apps/api and exports service symbol createGoal.
```

File chunks should not copy file contents. Hashes and sizes may be metadata but should not be retrieval prose unless a query explicitly targets scan integrity.

### Symbol Summary Chunks

Symbol summaries should identify the symbol, kind, export kind and path:

```text
Symbol createGoal is a named exported service in apps/api/src/services/goalService.ts.
```

If confidence is below a future persistence threshold, the symbol should be omitted from chunks or marked as low-confidence metadata.

### Edge Summary Chunks

Edge summaries should express a single relationship:

```text
File apps/api/src/routes/goals.ts imports apps/api/src/services/goalService.ts from a static relative import.
```

Edge chunks should include `edgeType`, source, target, confidence and evidence metadata.

### Route Summary Chunks

Route chunks should be generated from route symbols with literal method/path metadata:

```text
Route POST /api/goals is defined in apps/api/src/routes/goals.ts by symbol registerGoalRoutes.
```

Route chunks should not infer request/response behavior unless the scanner artifact contains explicit evidence and confidence.

### Migration Summary Chunks

Migration chunks should summarize literal SQL table operations detected by the scanner:

```text
Migration infra/sql/0001_create_goals.sql contains CREATE TABLE goals.
```

They must not imply database state, run migrations or infer complex schema semantics.

### Test Coverage Summary Chunks

Test chunks should describe scanner evidence, not behavioral assurance:

```text
Test file apps/api/src/__tests__/goalService.test.ts imports apps/api/src/services/goalService.ts.
```

Avoid claims such as "fully covers" unless a future evaluator proves that claim through explicit test coverage data.

### Doc Relationship Chunks

Doc/spec/ADR chunks should describe explicit relationships:

```text
Spec docs/specs/GOALS_SPEC.md documents apps/api/src/routes/goals.ts based on an explicit repository path mention.
```

Low-confidence documentation associations should not override code-derived relationships.

## Retrieval Behavior

Future retrieval may return Code Graph chunks for questions about repository structure, such as:

- which route file defines an API route,
- which service a route imports,
- which test imports a service,
- which migration touches a table,
- which spec or ADR documents a module,
- what files export a symbol.

Answerability must treat Code Graph chunks as derived context. If Code Graph chunks contradict newer code, specs or ADRs, the primary sources should win. Future answerability logic should be conservative:

- Code Graph cannot be the sole authority when primary source context is available and conflicts,
- low-confidence chunks must not relax abstention,
- warnings must not count as strong positive evidence,
- stale artifacts should lower confidence or trigger abstention,
- fallback lexical retrieval must continue to work when Code Graph context is absent, deleted, stale or unembedded.

Code Graph retrieval should complement existing `lexical`, `mock_vector` and `hybrid` modes. It must not require pgvector, real semantic embeddings, external providers or GraphRAG.

## Data Policy Interaction

Code Graph artifacts should avoid full file contents and should not include secrets. Future ingestion still must apply Context Data Policy before chunks are persisted or embedded.

Rules:

- artifact-derived text must be scanned before chunk persistence,
- restricted content must be blocked or excluded,
- secret or confidential findings must be redacted before embeddings,
- sensitive findings must not store matched values,
- generated chunks must not include full source content,
- warnings and metadata must not expose raw sensitive values,
- external providers remain prohibited without a separate approval and data handling policy.

If the artifact unexpectedly contains restricted or secret-like content, future ingestion should block the affected document or exclude affected records before chunking. Redaction must happen before mock, local or future vector embeddings.

## Retention and Deletion Interaction

Code Graph context is derived and regenerable. Future implementation must fit existing retention and deletion behavior:

- `code_graph` sources should be soft-deletable like other context sources,
- deleting the source deletes or soft-deletes derived documents, chunks and embeddings from active search,
- deleting the source does not delete repository code,
- regeneration from a newer artifact should create a new source or new versioned documents rather than mutating historical retrieval snapshots silently,
- quota limits must apply to generated documents, chunks, embeddings and retrieval logs,
- hard delete, if allowed by active policy, should remove only derived context rows and retain audit behavior.

Historical retrieval snapshots may still reference chunks selected before deletion, following existing retention caveats. Future implementation should make scan run and commit metadata visible enough to diagnose stale retrieval context.

## Evaluation Plan

Future Code Graph ingestion should connect to the existing retrieval-eval approach before runtime integration.

Recommended fixture questions:

- route questions, such as "which file defines POST /api/goals",
- service relationship questions, such as "which route imports createGoal",
- test questions, such as "which tests import goalService",
- migration questions, such as "which migration creates the goals table",
- doc relationship questions, such as "which ADR documents goalRepository",
- symbol questions, such as "where is GoalInput exported",
- no-answer questions for missing routes, symbols or tables,
- ambiguous questions with overlapping names,
- stale artifact cases where code/spec context is newer than the scan,
- warning cases where unsupported dynamic imports must not support a confident answer.

Evaluation should measure whether expected Code Graph chunks are retrieved without weakening existing no-answer and abstention behavior. It should use synthetic fixtures first and should not require PostgreSQL extensions, real models, external providers, GraphRAG or LLM-as-judge.

## Safety Constraints

Future Code Graph ingestion must preserve these limits:

- no code execution,
- no package script execution during ingestion,
- no database migrations,
- no runtime behavior changes unless a later milestone explicitly implements them,
- no external providers,
- no real models,
- no required pgvector,
- no reading outside the repository,
- no following symlinks outside the repository,
- no ingestion of unvalidated artifacts,
- no use of Code Graph as the only authority,
- no GraphRAG,
- no agentic actions,
- no filesystem source adapter exposed through product APIs without a separate spec,
- no full source-code ingestion as context chunks.

## Open Questions

- Should artifact ingestion be manual, a local tooling command or an API-mediated operation?
- Should future documents be grouped by file, by semantic group or by scan run?
- How should scans be versioned by commit and scanner version?
- How should stale artifacts be detected and surfaced during answerability?
- Should warnings become chunks, metadata only or retrieval-excluded audit records?
- How should scanner confidence affect ranking and answerability thresholds?
- What confidence threshold should be required for docs/spec/ADR relationship chunks?
- How should duplicate chunks be detected across regenerated artifacts?
- Should regeneration replace active Code Graph context or create a new source per scan?
- What evaluation coverage is required before runtime `load_context` can include Code Graph context?
- When is the project ready to reconsider GraphRAG after normal RAG over Code Graph chunks is evaluated?
